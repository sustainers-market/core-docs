Overview

The Core service provides identity, transaction, and accounting functionality for the Sustainer's Market.

To interact with the network, issue Commands to the endpoints described in the operations section, and listen to events published to a webhook.

The service is composed of the role, principle, auth-token, person-account, service-account, service, webhook, and challenge domains.

| role

A role is a grouping of scopes. It lets you create meaningful labels that characterize what a group of access rights mean.

For example, an Admin role can be characterized by scopes that give access to all resources.

Schema

{
  root: "some-uuid",
  name: "Some Role",
  scopes: [
    { 
      domain: "service", 
      priviledges: ["create"]
    },
    { 
      domain: "challenge", 
      priviledges: ["verify"],
      root: "<some-challenge-root>"
    }
  ]
}

The role in the example above has access to services that allow the service:create or challenge:verify scopes.

| principle

A principle can create auth-token's to interact with the network according to the roles it has been given.

Schema

  {
    root: "some-uuid"
    roles: [
      "<some-role-root>",
      "<someother-role-root>",
    ]
  }

| auth-token

An auth-token represents a JWT used as authentication and authorization for operations on a principle's behalf.

Schema

  {
    root: "some-uuid",
    token: "<the jwt token>",
  }

Commands

create

| challenge

A challenge represents an instance of federated identity authorization by verifying a code sent to a trusted channel (like a phone number).

Schema

  {
    root: "some-uuid",
    code: 123456,
    hash: "<some-data-hash>"
  }

Commands

create

| person-account

A person-account represents a person with a unique phone number interacting with the network through a principle.

Schema

  {
    root: "some-uuid",
    phone: "+11234567890",
    phoneVerified: true,
    id: "some-chosen-unique-id",
    name: {
      short: "Some",
      full: "Some Full Name"
    },
    principle: "<some principle>"
  }

create

| service-account

A service-account represents a service interacting with the network through a principle.

Schema

  {
    root: "some-uuid",
    id: "some-auto-generated-id",
    domain: "core.sustainers.network",
    secret: 1234567890,
    principle: "<some principle>"
  }

| service

A service represents a business on the network.

Schema

  {
    root: "some-uuid",
    name: "Some Service Name",
    id: "some-chosen-unique-in"
  }

create

| webhook

A webhook represents an endpoint to publish Events to. For example, a did-create.service event will get published as a result of issuing the create.service command.

Schema

  {
    root: "some-uuid",
    name: "Some Name",
    url: "https://some.endpoint.to.send/events/to"
  }

Servers

https://core.sustainers.network http

Use this server to interact with the production network and create real transactions.

https://core.sandbox.sustainers.network http

Use this server to access sandbox data. All transactions will be faked.
